# 云端会话同步方案（设计文档）

> 状态：**设计稿，未实现**。本文涉及的所有 worker.js 改动只停留在文档里，等静怡另行安排再落地。
> 目标：把聊天会话（`emet.chatSessions`）从「只在本机 localStorage」升级为「云端持久化 + 多设备同步」，
> 同时**与现有记忆数据完全隔离**，不污染记忆库。

当前现状：会话只存浏览器 localStorage `emet.chatSessions`（见 `src/utils/sessions.js`）。
卡片 3a 已实现「导出/导入 JSON」做手动迁移；本方案是其云端自动版。

> **动机（为什么 localStorage 不够）**：静怡现在有三个互相隔离的存储仓——①电脑浏览器
> ②手机 Safari 标签页 ③手机"添加到主屏幕"的 PWA 图标。其中 ②③ 在**同一台手机**上都不互通：
> iOS 把主屏幕 PWA 当成独立 app、有自己的存储沙箱，与 Safari 完全隔离。localStorage 是本机的，
> 天生跨不了设备、在 iOS 上甚至跨不了"同机不同入口"。**唯一解是云端做唯一真相源**，每个仓各自
> 与云端 pull/push。

> **决策已定（见 §7）**：多设备 → 增量同步 + **消息级合并** + 删除用**墓碑** + **自动后台同步**。

## 1. 数据结构

### 1.1 后端 KV（沿用现有 `env.MEMORY` 命名空间，独立前缀隔离）

现有记忆数据用前缀 `mem:` / `mom:` / `diary:` / `msg:` / `handoff:` / `idea:` / `game:`。
会话**新增独立前缀 `chat:`**，与记忆数据物理隔离在同一 KV 里：

```
chat:<sessionId>   →   {
  id:         string,        // 'c' + 创建时的 epoch ms（前端已有）
  title:      string,
  created_at: ISO 字符串,
  updated_at: ISO 字符串,    // 已在 3a 引入，做 last-write-wins 的依据
  messages:   [ { mid, ts, role, content, thinking?, tools? } ],
                            // mid=消息唯一id，ts=创建时间戳；做消息级并集合并的依据
                            //（前端同步层会给新消息补上，老消息无 mid 时退化为"长者胜"）
  deleted?:   boolean        // 软删除墓碑（见 §5）：删除写 deleted:true 而非真删，跨设备传播
}
```

> 之所以**一会话一个 KV key**（而非把所有会话塞进一个大 value）：
> - 单条会话可独立 upsert，增量同步只传变化的会话，不必每次搬运全量。
> - 避免单 value 体积膨胀（消息可能很长）触及 KV value 上限。
> - 与现有记忆"一条一 key"的模式一致。

### 1.2 隔离保证

- `chat:` 前缀与任何记忆前缀不重叠；`kvListByPrefix(env, 'chat:')` 只会捞到会话。
- `GET /api/data`（记忆全量）**不包含** `chat:`，保持记忆视图纯净。
- 所有 MCP 工具（`executeTool`）只读写 `mem:/mom:/diary:/...`，**不触碰 `chat:`**。
- `backup_export` 是否纳入会话：建议**单独**走会话导出（§4 的 `GET /api/chat`），不混进记忆备份。

## 2. 新增路由（worker.js，待实现）

全部挂在 `/api/` 下，**自动复用现有 `X-Admin-Key` 鉴权闸门**（`routeRequest` 第 5310 行已对 `/api/*` 统一拦截），无需额外鉴权代码。

| 方法 | 路径 | 用途 | 返回 |
|---|---|---|---|
| GET | `/api/chat` | 拉全部会话（含 messages） | `{ sessions: [...] }` |
| GET | `/api/chat?since=<ISO>` | 增量：只回 `updated_at > since` 的会话 | `{ sessions: [...], server_time }` |
| PUT | `/api/chat/:id` | upsert 单条会话（body = 会话对象） | `{ success, item }` |
| DELETE | `/api/chat/:id` | 删除单条（或写墓碑，见 §5） | `{ success }` |
| POST | `/api/chat/sync` | **批量**：上传本地会话数组，服务端按 `updated_at` 合并，回合并后的全量或增量 | `{ sessions, server_time }` |

> 路由风格与现有 `handleAPIv2` 一致（`path.match(/^\/api\/chat\/(.+)$/)`）。
> 写操作天然受 `method !== "GET" && checkAuth` 二次兜底保护。

### 2.1 worker.js 改动（最终代码，交静怡应用）

**块 A —— 合并函数**（顶层，建议贴在 `checkLockBeforeDelete` 之后）：

```js
// ─── 会话合并（消息级并集 + 会话级 last-write-wins）───
function mergeMessages(a = [], b = []) {
  const hasMid = (arr) => arr.length > 0 && arr.every((m) => m && m.mid);
  if (hasMid(a) && hasMid(b)) {
    const map = new Map();
    for (const m of [...a, ...b]) {
      const ex = map.get(m.mid);
      // 同 mid 取 content 更长的一份（流式可能某端只存了半截）
      if (!ex || (m.content || "").length > (ex.content || "").length) map.set(m.mid, m);
    }
    return [...map.values()].sort((x, y) => {
      const xt = x.ts || "", yt = y.ts || "";
      return xt < yt ? -1 : xt > yt ? 1 : 0;
    });
  }
  return a.length >= b.length ? a : b; // 无 mid 退化：取更长的一方
}

function mergeSession(a, b) {
  if (!a) return b;
  if (!b) return a;
  const newer = (a.updated_at || "") >= (b.updated_at || "") ? a : b;
  const older = newer === a ? b : a;
  return {
    ...newer,                                          // 会话级字段（标题/删除标记）取较新
    created_at: older.created_at || newer.created_at,  // 创建时间取较早
    messages: mergeMessages(a.messages || [], b.messages || []),
  };
}
```

**块 B —— 路由**（`handleAPIv2` 内，贴在 `/api/unlink` 段之后、信件段之前）：

```js
// ─── 会话云同步（chat: 前缀，与记忆数据隔离）───
if (path === "/api/chat" && method === "GET") {
  const since = url.searchParams.get("since");
  let sessions = await kvListByPrefix(env, "chat:");
  if (since) sessions = sessions.filter((s) => (s.updated_at || "") > since);
  return jsonResponse({ sessions, server_time: now() });
}
// 批量对账（必须在 :id 正则之前）
if (path === "/api/chat/sync" && method === "POST") {
  const body = await request.json();
  const incoming = Array.isArray(body.sessions) ? body.sessions : [];
  for (const s of incoming) {
    if (!s || !s.id) continue;
    const existing = await kvGet(env, "chat:" + s.id);
    await kvPut(env, "chat:" + s.id, mergeSession(existing, s));
  }
  const all = await kvListByPrefix(env, "chat:");
  return jsonResponse({ sessions: all, server_time: now() });
}
const chatMatch = path.match(/^\/api\/chat\/(.+)$/);
if (chatMatch && method === "PUT") {
  const id = chatMatch[1];
  const body = await request.json();
  const existing = await kvGet(env, "chat:" + id);
  const merged = mergeSession(existing, { ...body, id });
  await kvPut(env, "chat:" + id, merged);
  return jsonResponse({ success: true, item: merged });
}
if (chatMatch && method === "DELETE") {
  const id = chatMatch[1];
  const existing = await kvGet(env, "chat:" + id);
  const tomb = { ...(existing || { id }), id, deleted: true, updated_at: now() };
  await kvPut(env, "chat:" + id, tomb); // 墓碑，不真删
  return jsonResponse({ success: true });
}
```

## 4. 前端同步策略

### 4.1 已定：增量同步 + 消息级合并（last-write-wins 仅用于会话级字段）

会话级字段（标题、删除标记）按 `updated_at` last-write-wins；**会话内的 messages 走消息级并集**，
两端聊同一段对话也不丢消息。合并函数 `mergeSession(a,b)` 前后端同构：

- **会话级字段**：取 `updated_at` 较新的一方（标题 / 删除标记等）。
- **messages**：按 `mid` 并集去重（同 mid 取 content 更长的一份，应对流式半截）、按 `ts` 排序；
  消息无 mid（老数据）时退化为"messages 更长的一方胜"（append-only 下通常一方是另一方前缀）。
- **拉取（pull）**：`GET /api/chat?since=<上次同步时间>` → 用 `mergeSession` 并入本地。
- **推送（push）**：本地会话变更时 `PUT /api/chat/:id`，服务端也用 `mergeSession` 与已存版本合并后落库。
- **对账（sync）**：`POST /api/chat/sync` 上传本地全量，服务端逐条 merge 落库、回权威全量。

### 4.2 备选：全量覆盖

每次同步整包上传/下载、直接覆盖。实现最简，但：
- 多设备并发会互相覆盖丢消息；
- 流量随会话增长线性膨胀。
**仅适合"单设备 + 偶尔备份"**，不推荐作为长期方案。

### 4.3 结论

**选 4.1 增量同步 + 消息级合并**（拒绝整条覆盖，避免两端聊同一会话丢消息）。
`updated_at` 已在 3a 落地；前端同步层再为每条消息补 `mid`/`ts`，地基即齐。

### 4.4 同步触发时机（前端，待实现）

| 时机 | 动作 |
|---|---|
| 聊天页挂载 | pull（`GET /api/chat?since=<上次同步时间>`）合并入本地 |

| 设置页「立即同步」按钮 | `POST /api/chat/sync` 全量对账 |

本地仍保留 localStorage 作为离线缓存与失败兜底（在线失败不阻塞聊天）。

## 5. 删除同步（需拍板）

纯硬删（`DELETE` → `kvDelete`）在多设备下有"删除复活"问题：A 删了、B 还没同步，B 的 push 会把它又传回来。

两个选项：
- **A（简单）**：硬删。单设备/低频场景够用，接受极小概率复活。
- **B（稳健）**：**墓碑**——删除写 `{ deleted:true, updated_at }` 而非真删，同步按 `updated_at` 传播删除意图，定期清理老墓碑。多设备推荐。

**已定：B 墓碑**（多设备必须——否则一端删了、另一端没同步，push 会把它又传回来）。
删除写 `{ deleted:true, updated_at:now }` 而非真删；pull 时客户端见到 `deleted:true` 即删本地对应会话。

## 6. 与卡片 3a 的关系

- 3a（已实现）：本地 ⇄ JSON 文件的**手动**导出/导入，离线迁移用。

- 二者**合并规则同源**（`updated_at` last-write-wins），3a 的 `importSessions` 可抽出复用。
- 文件格式 `format/version` 字段也建议沿用到云端 payload，便于未来结构升级时识别。
